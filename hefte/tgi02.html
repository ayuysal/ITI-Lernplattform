Kapitel 1

Höheres Sortieren
TEILE UND HERRSCHE (DIVIDE & CONQUER)

Das zentrale Metaprinzip für höhere Sortieralgorithmen:

  • Split-Phase: Aufteilen der Datenmenge in kleinere Teile
  • Merge-Phase: Zusammenfügen der Ergebnisse
  • Rekursion: Der Algorithmus ruft sich selbst mit kleineren Daten auf

Grundidee: Eine kleinere Liste sortiert sich leichter als eine große!

QUICKSORT

Prinzip: Sortierung in der Split-Phase (Teile & Herrsche + Rekursion).

  1. Wähle das mittlere Element als Pivot
  2. Bilde zwei Teillisten: A1 (≤ Pivot) und A2 (> Pivot)
  3. Rufe QuickSort rekursiv auf A1 und A2 auf
  4. Ergebnis: sortierte A1 + Pivot + sortierte A2

Beispiel: A = [13, 6, 7, 2, 9, 11, 4] → Pivot = 2 (Index 3)
A1 = [] → A2 = [13, 6, 7, 9, 11, 4] → rekursiv weitersortieren ...

Eigenschaft: Einer der am meisten verwendeten Sortieralgorithmen.

MERGESORT

Prinzip: Sortierung in der Merge-Phase. Split ist trivial (Liste mittig teilen).

  1. Teile Liste in der Mitte (ohne Vergleiche!)
  2. Sortiere beide Hälften rekursiv
  3. Merge: Vergleiche die Köpfe beider sortierter Listen, übernimm den kleineren

Beispiel Merge: S1 = [2, 4, 9] und S2 = [8, 10, 15]
→ Vergleiche 2 < 8 → Asort = [2] → 4 < 8 → [2, 4] → 8 < 9 → [2, 4, 8] → ...

HEAPSORT – GRUNDLAGEN

Heap = Wurzelbaum, bei dem jeder Vater größer als seine Söhne ist (Maximum-Heap).

Speicherung im Array: Ebene für Ebene, links nach rechts. Kinder von Index x stehen bei (2x+1) und (2x+2).

HeapSort-Algorithmus:

  1. Heapify: Unsortierte Liste → Heap umformen (alle Nichtblätter durchsickern lassen, von hinten)
  2. Maximum (Wurzel) entnehmen, ans Ende tauschen
  3. Durchsickern (siftDown): Neues Wurzelelement mit dem größten Sohn tauschen, bis Heap-Eigenschaft wiederhergestellt
  4. Wiederhole ab Schritt 2
RADIXSORT

Prinzip: Nicht „in situ"! Braucht zusätzlichen Speicher (Boxen/Fächer).

  • Elemente nach einem Kriterium (z.B. erste Ziffer) in Boxen einsortieren
  • Jede Box einzeln sortieren oder erneut in Unterboxen aufteilen
  • Beispiel Post: Briefe nach 1. PLZ-Ziffer in 10 Fächer → jedes Fach mit MergeSort sortieren

Voraussetzung: Überschaubare Anzahl sinnvoller Fächer, möglichst gleichmäßig gefüllt.

Vorteil: Wenn einsetzbar, schneller als alle anderen hier vorgestellten Algorithmen!

Kapitel 2

Auswählen und Selektieren
MAXIMUMSUCHE – SCHRITTWEISE VERBESSERUNG

Drei Ansätze zur Maximumsuche in unsortierten Listen:

  • MaxSearch1: Erst sortieren, dann letztes Element nehmen → funktioniert, aber uneffektiv
  • MaxSearch2: BubbleSort 1. Iteration → Maximum blubbert ans Ende (n−1 Vergleiche + Tauschaktionen)
  • MaxSearch3: SelectionSort-Variante → nur merken, wo Maximum liegt (n−1 Vergleiche, keine Tauschaktionen)
MIN-MAX-SUCHE – VERGLEICHSANALYSE
MethodeVergleichePrinzip
MinMaxSearch1 (2× durchlaufen)2n − 2Erst Max suchen, dann Min suchen
MinMaxSearch2 (1× durchlaufen)2n − 2Min und Max gleichzeitig prüfen
MinMaxSearch3 (Tournament)3n/2 − 2Erst Paare bilden, dann getrennt suchen

Tournament-Prinzip: n/2 Paare bilden → n/2 Vergleiche. Dann unter Gewinnern Max suchen (n/2−1) + unter Verlierern Min (n/2−1) = 25% Ersparnis!

MAXIMUM + ZWEITBESTES (TOURNAMENT)

Wie im Sport: Paare bilden → Gewinner kommt weiter → Finale.

Der Zweitbeste muss irgendwann gegen den Sieger verloren haben!

Vergleiche: (n − 1) für Sieger + log₂(n) für Zweitbesten aus den Verlierern des Siegers.

Gesamt: n − 1 + log₂(n) statt 2n − 3 (naive Methode).

K-SELEKTION

Suche das k-größte Element, ohne komplett zu sortieren:

  • QuickSelection: Wie QuickSort, aber nur die Teilliste weiterverfolgen, die das gesuchte Element enthält
  • HeapSelection: Heap bauen, dann (k−1)-mal Maximum entnehmen → k-tes steht oben
SICH SELBST ANORDNENDE LISTEN

Häufig zugegriffene Elemente wandern nach vorne → schnellerer Zugriff über Zeit.

StrategiePrinzipPro/Contra
MF (Move to Front)Zugegriffenes Element direkt an den AnfangSchnell, aber ein Fehlgriff stört stark
T (Transpose)Element tauscht mit Vorgänger (1 Position vor)Behutsam, aber kann blockieren
FC (Frequency Count)Zähler pro Element, sortiert nach HäufigkeitAm effizientesten, aber aufwendiger

Anwendung: Caches, Zugriffsoptimierung im Millisekundentakt.

Kapitel 3

Hashing
HASHFUNKTION – DEFINITION

Eine Hashfunktion h bildet Schlüsselwerte auf Indizes einer Hashtabelle ab:

h : K → {0, 1, …, m−1}

Dabei ist m die Größe der Hashtabelle. Anforderungen an eine gute Hashfunktion:

  • Schnell berechenbar
  • Surjektiv – jeder Tabellenindex wird von mindestens einem Schlüssel getroffen
  • Gleichmäßige Verteilung – Schlüssel möglichst gleichmäßig über Tabelle verteilen
HASHVERFAHREN
VerfahrenFormelHinweis
Divisionsresth(k) = k % mm sollte Primzahl sein (nicht 10!)
Multiplikationh(k) = ⌊(ak − ⌊ak⌋) · m⌋a nahe Goldenem Schnitt φ ≈ 0,618
FaltungZiffern aufteilen, addieren, dann Divisionsrestz.B. 955323 → 955+323 = 1278 → 1278%17 = 3
BasiswechselDezimal → Oktal → als Dezimal lesen → Divisionsrestz.B. 3603 → 7023₈ → 7023%17 = 2
KOLLISION – h(k₁) = h(k₂)

Zwei verschiedene Schlüssel landen auf dem gleichen Index. Unvermeidlich, da |K| >> m.

GESCHLOSSENES VERFAHREN

Idee: Überlaufbereich außerhalb der Tabelle.

  • Bei Kollision: Zeiger auf externe verkettete Liste
  • Neue Elemente werden dort angehängt
  • = Kollisionsauflösung durch Verkettung

Nachteil: Zusätzliche Speicherverwaltung für Zeigerstrukturen.

OFFENES VERFAHREN

Idee: Freien Platz innerhalb der Tabelle suchen.

  • Lineares Sondieren: Nächsten freien Platz rechts suchen (1, 2, 3, ...)
  • Quadratisches Sondieren: Schrittweite 1, 4, 9, 16, ... → weniger Klümpchenbildung

Vorteil: Kein Zusatzspeicher. Nachteil: Klümpchenbildung bei linearem Sondieren.

Kapitel 4

Pattern Matching & Schnelles Rechnen
MATCHING – TEXTSUCHE

Matching = Suche eines Patterns (Suchstring) als Teilstring in einem Scantext.

Brute-Force: Pattern an jede Position im Scantext anlegen und alle Zeichen vergleichen.

Funktioniert, aber bei langem Text und langem Pattern sehr langsam.

KMP-MATCHING (KNUTH-MORRIS-PRATT)

Kernidee: Bei Mismatch muss das Pattern nicht immer nur um 1 Position verschoben werden. Die Stelle des Mismatches verrät, wie weit man springen kann.

  • next-Array: Wird vorab für das Pattern berechnet
  • Bei Mismatch an Position j: setze pi = next[j]
  • Scantext wird nur einmal durchlaufen (kein Zurückspringen!)

Beispiel: S = "10001000000101010", p = "1000000"
Mismatch an Position 4 → nächster möglicher Match erst ab Position 4 selbst → 4 Stellen springen statt 1!

BOYER-MOORE-MATCHING

Kernidee: Von hinten scannen! Drei Regeln:

  1. Nur bei Match rückwärts weitergehen
  2. Bei Mismatch: Pattern so verschieben, dass das letzte Vorkommen des Mismatch-Zeichens im Pattern über dem aktuellen Zeichen liegt
  3. Wenn das Zeichen gar nicht im Pattern vorkommt → komplettes Pattern überspringen

Vorteil: Bei natürlichsprachlichen Texten sehr schnell, oft wird das ganze Wort übersprungen!

RABIN-KARP-MATCHING

Kernidee: Hashing zur Textsuche!

  1. Berechne Hash des Patterns
  2. Schiebe ein Fenster (gleiche Länge wie Pattern) über den Scantext
  3. Berechne Hash des aktuellen Fensters
  4. Nur wenn Hash übereinstimmt → genauen Vergleich durchführen

Beispiel: p = 841, h = x % 11 → hash(841) = 5. Im Scantext nur bei hash = 5 genau prüfen.

SCHNELLES RECHNEN – VERBESSERTE MULTIPLIKATION

Zwei n-stellige Zahlen: Standardmethode braucht n² Multiplikationen.

Teile & Herrsche: Zahlen mittig teilen (z.B. 2981 → 29 und 81).

Trick: Statt 4 Teil-Multiplikationen nur 3 nötig durch geschickte Umrechnung:

  • Berechne wy, xz und r = (w+x)(y+z) − wy − xz
  • Ergebnis: 10000·wy + 100·r + xz

Ersparnis: 25% weniger Multiplikationen, erkauft durch zusätzliche Additionen/Subtraktionen.

Strassen-Algorithmus: Matrizenmultiplikation von n³ auf n^2,81 reduziert.

Kapitel 5

Analyse von Algorithmen
ZEITKOMPLEXITÄT T(n)

Die Zeitkomplexität beschreibt die Abarbeitungszeit eines Algorithmus als Funktion der Eingabelänge n.

Jede Codezeile hat eine konstante Ausführungszeit ci. Die Gesamtzeit ergibt sich aus der Summe aller Ausführungen.

BEISPIELE FÜR ZEITKOMPLEXITÄT
Code-TypStrukturT(n)
Sequentiell (keine Schleife)x = x + 2d (konstant)
Einfache Schleife (feste Grenze)while (i < 11)d (konstant)
Einfache Schleife (Eingabe n)for i in 0..n-1d₁·n + d₂ (linear)
Verschachtelte SchleifenBubbleSortd₁·n² + d₂·n + d₃ (quadratisch)
BUBBLESORT – ZEITKOMPLEXITÄT

Äußere Schleife: n Durchläufe. Innere Schleife: n−1 Durchläufe (jeweils!).

TBubbleSort(n) = d₁·n² + d₂·n + d₃

Praxistest: Verdopplung von n (5000→10000) → vierfache Laufzeit. Bestätigt quadratisches Wachstum!

INSERTIONSORT – BEST VS. WORST CASE

Die innere Schleife hängt von den Daten ab (wie weit muss der key nach links?).

FallBedingungT(n)
Best CaseBereits sortiert (ti = 0 für alle i)d₁·n + d₂ → linear
Worst CaseUmgekehrt sortiert (ti = i für alle i)d₁·n² + d₂·n + d₃ → quadratisch

Bei Worst Case: Gaußsche Summenformel → Σi = n(n+1)/2 → quadratisch.

BOTTLENECK-PROBLEM

Der Exponent bestimmt das Wachstum, nicht die Koeffizienten!

T₁(n) = 6n + 3 vs. T₂(n) = n² + n + 1 → T₂ wächst für große n immer schneller, egal wie groß die Konstanten von T₁ sind.

Konsequenzen für Programmierer:

  • Schleifenlänge reduzieren
  • Verschachtelungstiefe reduzieren! (eine 4-fach verschachtelte Schleife → n⁴)
  • Bottlenecks finden: Die eine Stelle mit dem höchsten Exponenten optimieren → Code-Refactoring

Kapitel 6

Klassifikation von Algorithmen
O-NOTATION (LANDAU-SYMBOLE)

O(f) = Menge aller Funktionen, die nicht schneller als f wachsen (Upper Bound).

Formal: g ∈ O(f) ⟺ es gibt n₀ und c, sodass für alle n ≥ n₀: g(n) ≤ c · f(n)

Beispiele für f(n) = n²:

  • 3n + 5 ∈ O(n²) ✓ (wächst langsamer)
  • 0,5n² ∈ O(n²) ✓ (Konstante egal)
  • n³ ∉ O(n²) ✗ (wächst schneller)

Schreibweise: Statt g ∈ O(n²) schreibt man oft g = O(n²) (formal unsauber, aber üblich).

WORST CASE & AVERAGE CASE
BegriffBedeutungBeispiel InsertionSort
Best CaseGünstigste EingabeO(n) – bereits sortiert
Worst CaseUngünstigste EingabeO(n²) – umgekehrt sortiert
Average CaseDurchschnittliche EingabeO(n²) – zufällige Reihenfolge
MASTERTHEOREM

Für rekursive Algorithmen der Form: T(n) = a · T(n/b) + f(n)

  • a = Anzahl rekursiver Aufrufe
  • b = Faktor der Problemverkleinerung
  • f(n) = Aufwand außerhalb der Rekursion

MergeSort: T(n) = 2·T(n/2) + n → a=2, b=2 → O(n log n)

QuickSort: Average Case ebenfalls O(n log n), aber Worst Case O(n²)!

P VS. NP – DAS 1-MIO-$-PROBLEM

P: Probleme, die in polynomieller Zeit lösbar sind (O(n^k) für festes k).

NP: Probleme, deren Lösung in polynomieller Zeit überprüfbar ist.

Offene Frage: Ist P = NP? Falls ja: jedes Problem, dessen Lösung schnell überprüfbar ist, wäre auch schnell lösbar.

Preisgeld: 1.000.000 US-$ (Clay Mathematics Institute, Millennium Prize)
